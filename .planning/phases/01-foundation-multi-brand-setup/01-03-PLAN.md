  <name>Task 1: Build admin API routes for brands, locations, staff, and ERP sync</name>
  <files>
    src/app/api/brands/route.ts, src/app/api/brands/[brandId]/route.ts,
    src/app/api/locations/route.ts, src/app/api/locations/[locationId]/route.ts,

  </files>
  <action>
    **Brand API** (src/app/api/brands/route.ts):
    - GET: List all brands (admin only — check role from session). Return array of brands.
    - POST: Create brand. Accept {name, logoUrl?, address?, taxId?, serviceChargePct?, vatPct?}. Validate with zod. Return 201 with created brand.
    - IMPORTANT: In Next.js 16, params are async. Use `const { brandId } = await params;`

    **Brand Detail** (src/app/api/brands/[brandId]/route.ts):
    - GET: Return single brand with its locations
    - PUT: Update brand fields. Validate with zod.
    - DELETE: Soft-delete (set isActive=false) or hard delete if no dependent data

    **Location API** (src/app/api/locations/route.ts):
    - GET: List locations. Admin sees all; manager sees own location only (filter by session.locationId)
    - POST: Create location with brandId and settings JSON. Admin only.

    **Location Detail** (src/app/api/locations/[locationId]/route.ts):
    - GET: Single location with settings
    - PUT: Update location settings (service charge %, printer config, etc.)

    **Staff API** (src/app/api/staff/route.ts):
    - GET: List staff. Admin sees all; manager sees own location staff.
    - POST: Create staff user. Accept {name, role, brandId, locationId, pin?, email?, password?}. Hash PIN/password with bcryptjs before storing. Admin only.

    **ERP Sync Types** (src/types/erp.ts):
    - ERPCategory, ERPMenuItem, ERPModifier, ERPSyncResponse types
    - SyncResult type: { success, itemsSynced, categoriesSynced, errors? }

    **ERP Sync Service** (src/lib/erp-sync.ts):
    - syncMenuFromERP(brandId: string): Promise<SyncResult>
    - Uses adapter pattern: fetchERPMenu(brandId) is a separate function that can be swapped for real ERP later
    - For Phase 1: fetchERPMenu returns mock data (realistic Japanese ramen menu for "A RAMEN", burger menu for "Burger Lab")
    - Wraps entire upsert in db.transaction() to prevent partial failures (per research pitfall #5)
    - Upserts categories, items, modifier groups, modifier options using onConflictDoUpdate on erpId+brandId
    - Logs to sync_logs table (start time, status, items synced, errors)
    - Per user decision: manual-only sync (admin clicks button), no polling/webhook yet

    **Sync Trigger** (src/app/api/sync/route.ts):
    - POST: Accept {brandId}. Admin only. Calls syncMenuFromERP. Returns SyncResult.

    **Tests**:
    - src/lib/__tests__/erp-sync.test.ts: Test mock ERP data shape, sync result structure
    - src/app/api/__tests__/brands.test.ts: Test brand API request/response shapes with zod validation

    All API routes must:
    - Read session from cookie using verifySession() from src/lib/auth.ts
    - Check permissions using hasPermission() from src/lib/permissions.ts
    - Return 401 if no session, 403 if insufficient permissions
    - Use zod for input validation, return 400 with error details on invalid input
    - Handle errors with try/catch, return appropriate status codes
  </action>
